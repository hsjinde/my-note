# LLM Wiki

一種使用 LLM 建立個人知識庫的模式。

這是一個概念文件，目的是讓你複製貼上到自己的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目標是傳達高層次的概念，但你的 Agent 會與你協作，建構出具體的實作細節。

## 核心概念

大多數人使用 LLM 處理文件的經驗類似 RAG：上傳一批文件，LLM 在查詢時檢索相關片段，然後生成答案。這可行，但 LLM 每次回答問題時都是從頭重新發現知識。沒有任何累積。問一個需要綜合五份文件的細緻問題，LLM 每次都得找出並拼湊相關片段。沒有東西被建立起來。NotebookLM、ChatGPT 檔案上傳，以及大多數 RAG 系統都是這樣運作的。

這裡的概念不同。LLM 不只是從原始文件檢索，而是**漸進式地建立並維護一個持久化的 Wiki**——一個結構化、互相連結的 Markdown 檔案集合，位於你與原始資料之間。當你加入新來源時，LLM 不只是為後續檢索建立索引。它會閱讀、提取關鍵資訊，並整合進現有的 Wiki——更新實體頁面、修訂主題摘要、標記新資料與舊觀點矛盾之處、強化或挑戰正在演進的綜合論述。知識被編譯一次，然後*保持更新*，而不是每次查詢時重新推導。

這就是關鍵差異：**Wiki 是一個持久、持續累積的產物。**交叉引用已經就位。矛盾已經被標記。綜合論述已經反映了你讀過的所有內容。Wiki 隨著你加入的每個來源和提出的每個問題而變得更豐富。

你從來（或很少）自己寫 Wiki——LLM 撰寫並維護全部內容。你負責尋找來源、探索和提出好問題。LLM 處理所有繁瑣工作——摘要、交叉引用、歸檔和記錄，這些讓知識庫隨著時間真正有用。實務上，我會在一邊開著 LLM Agent，另一邊開著 Obsidian。LLM 根據我們的對話進行編輯，我即時瀏覽結果——追蹤連結、查看圖譜視圖、閱讀更新後的頁面。Obsidian 是 IDE；LLM 是程式設計師；Wiki 是程式碼庫。

這可以應用在許多不同場景。幾個例子：

- **個人**：追蹤自己的目標、健康、心理、自我成長——歸檔日記條目、文章、Podcast 筆記，逐步建立一個關於自己的結構化圖像。
- **研究**：在數週或數月內深入一個主題——閱讀論文、文章、報告，逐步建立一個全面的 Wiki，附帶演進中的論點。
- **閱讀一本書**：隨著進度歸檔每一章，建立角色、主題、情節線索頁面，以及它們之間的關聯。讀完後你會有一個豐富的伴隨 Wiki。可以想像像是 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 這樣的粉絲 Wiki——數千個互相連結的頁面，涵蓋角色、地點、事件、語言，由志願者社群多年建立。你可以個人在閱讀過程中建立類似的東西，由 LLM 處理所有交叉引用和維護。
- **商業/團隊**：由 LLM 維護的內部 Wiki，由 Slack 討論串、會議記錄、專案文件、客戶通話提供資料。可能有人類參與審查更新。Wiki 保持最新，因為 LLM 處理團隊中沒有人想做的維護工作。
- **競爭分析、盡職調查、旅行規劃、課程筆記、興趣深入研究**——任何隨著時間累積知識並希望有組織而非零散的情況。

## 架構

有三個層次：

**原始資料（Raw sources）**——你精心挑選的來源文件集合。文章、論文、圖片、資料檔案。這些是不可變的——LLM 讀取它們，但絕不修改。這是你的事實來源。

**Wiki**——一個由 LLM 生成的 Markdown 檔案目錄。摘要、實體頁面、概念頁面、比較、總覽、綜合論述。LLM 完全擁有這個層次。它建立頁面、在新來源加入時更新它們、維護交叉引用、保持一切一致。你閱讀它；LLM 撰寫它。

**Schema**——一份文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告訴 LLM Wiki 的結構、慣例，以及在消化來源、回答問題或維護 Wiki 時應遵循的工作流程。這是關鍵的設定檔——它讓 LLM 成為一個有紀律的 Wiki 維護者，而不是一個通用聊天機器人。你與 LLM 隨著時間共同演化這份文件，摸索出適合你領域的作法。

## 操作

**Ingest（消化）。**你將一個新來源放進原始資料集，然後告訴 LLM 處理它。一個範例流程：LLM 閱讀來源、與你討論關鍵要點、在 Wiki 中撰寫摘要頁面、更新索引、更新整個 Wiki 中相關的實體和概念頁面，並在日誌中附加一筆記錄。單一來源可能涉及 10-15 個 Wiki 頁面。我個人偏好一次消化一個來源並保持參與——我閱讀摘要、檢查更新、引導 LLM 強調什麼重點。但你也可以一次批次消化多個來源，監督較少。由你發展適合自己風格的工作流程，並記錄在 Schema 中供未來 session 使用。

**Query（查詢）。**你對 Wiki 提出問題。LLM 搜尋相關頁面、閱讀它們，然後綜合出附帶引用的答案。答案可以根據問題類型採取不同形式——Markdown 頁面、比較表、投影片（Marp）、圖表（matplotlib）、Canvas。重要的洞見：**好的答案可以歸檔回 Wiki 成為新頁面。**你詢問的比較、分析、發現的關聯——這些都是有價值的，不應該消失在聊天記錄中。這樣你的探索就會像消化的來源一樣，在知識庫中持續累積。

**Lint（健康檢查）。**定期請 LLM 檢查 Wiki 健康狀況。尋找：頁面之間的矛盾、已被較新來源取代的過時觀點、沒有被其他頁面連結的孤立頁面、被提及但缺少獨立頁面的重要概念、遺漏的交叉引用、可以透過網路搜尋填補的資料缺口。LLM 擅長建議新的調查問題和尋找新來源。這讓 Wiki 在成長過程中保持健康。

## 索引與日誌

兩個特殊檔案幫助 LLM（和你）在 Wiki 成長時導航。它們有不同的用途：

**index.md** 是內容導向的。它是 Wiki 中所有內容的目錄——每個頁面附帶連結、一句摘要，以及可選的中繼資料如日期或來源數量。按類別組織（實體、概念、來源等）。LLM 在每次 ingest 時更新它。回答查詢時，LLM 先讀取索引來找到相關頁面，然後深入閱讀。在中等規模下（約 100 個來源、數百個頁面）運作意外地好，而且避免了對 embedding-based RAG 基礎設施的需求。

**log.md** 是時間順序的。它是一個只附加的操作記錄——ingest、查詢、lint 檢查。一個有用的技巧：如果每筆記錄以固定前綴開頭（例如 `## [2026-04-02] ingest | 文章標題`），日誌就可以用簡單的 Unix 工具解析——`grep "^## \[" log.md | tail -5` 顯示最後 5 筆記錄。日誌讓你看到 Wiki 的演進時間線，並幫助 LLM 了解最近做了什麼。

## 可選：CLI 工具

某個時間點你可能會想建立一些小工具，幫助 LLM 更有效率地操作 Wiki。Wiki 頁面的搜尋引擎是最明顯的一個——小規模時索引檔案就夠了，但隨著 Wiki 成長你需要真正的搜尋。[qmd](https://github.com/tobi/qmd) 是一個不錯的選擇：它是一個 Markdown 檔案的本機搜尋引擎，支援 BM25/向量混合搜尋和 LLM 重新排序，全部在本機裝置上執行。它同時有 CLI（LLM 可以 shell out 呼叫）和 MCP 伺服器（LLM 可以將其作為原生工具使用）。你也可以自己建構更簡單的東西——LLM 可以幫助你隨著需求產生，用 vibe-coding 寫一個簡單的搜尋指令稿。

## 提示與技巧

- **Obsidian Web Clipper** 是一個瀏覽器擴充功能，將網頁文章轉換為 Markdown。對於快速將來源放進原始資料集非常有用。
- **將圖片下載到本機。**在 Obsidian 設定 → 檔案與連結，將「附件資料夾路徑」設為固定目錄（例如 `raw/assets/`）。然後在設定 → 快速鍵，搜尋「下載」找到「下載目前檔案的附件」並綁定快速鍵（例如 Ctrl+Shift+D）。剪藏文章後，按下快速鍵，所有圖片就會下載到本機磁碟。這是可選的但很實用——讓 LLM 能直接查看和引用圖片，而不是依賴可能失效的 URL。請注意，LLM 無法一次性地讀取包含內嵌圖片的 Markdown——解決方法是讓 LLM 先讀取文字，然後單獨查看部分或全部的引用圖片以獲取額外脈絡。這有點笨拙但效果還不錯。
- **Obsidian 的圖譜視圖**是查看 Wiki 形狀的最佳方式——哪些頁面與哪些頁面關聯、哪些頁面是樞紐、哪些是孤立頁面。
- **Marp** 是基於 Markdown 的投影片格式。Obsidian 有對應的插件。對於直接從 Wiki 內容生成簡報非常有用。
- **Dataview** 是 Obsidian 插件，可以對頁面 frontmatter 執行查詢。如果你的 LLM 在 Wiki 頁面加入 YAML frontmatter（標籤、日期、來源數量），Dataview 可以生成動態表格和清單。
- Wiki 就是一個 Markdown 檔案的 git 儲存庫。你可以免費獲得版本歷史、分支和協作功能。

## 為什麼這有效

維護知識庫最繁瑣的部分不是閱讀或思考——而是記錄工作。更新交叉引用、保持摘要最新、標記新資料與舊觀點矛盾之處、維持數十個頁面之間的一致性。人類放棄 Wiki 是因為維護負擔增長得比價值更快。LLM 不會感到無聊、不會忘記更新交叉引用，可以在一次操作中處理 15 個檔案。Wiki 保持維護是因為維護成本趨近於零。

人類的工作是策劃來源、引導分析、提出好問題，以及思考這一切的意義。LLM 的工作是其他一切。

這個概念在精神上與 Vannevar Bush 的 Memex（1945）相關——一個個人策劃的知識儲存，文件之間有關聯軌跡。Bush 的願景更接近這個概念，而不是網路最終變成的樣子：私有的、積極策劃的，文件之間的連結與文件本身一樣有價值。他未能解決的部分是「誰來做維護」。LLM 處理了這個問題。

## 備註

這份文件刻意保持抽象。它描述的是概念，而不是特定實作。確切的目錄結構、Schema 慣例、頁面格式、工具選擇——所有這些都取決於你的領域、偏好和選擇的 LLM。以上提到的所有內容都是可選和模組化的——選擇有用的，忽略不需要的。例如：你的來源可能只有文字，所以你完全不需要圖片處理。你的 Wiki 可能夠小，索引檔案就足夠了，不需要搜尋引擎。你可能不在乎投影片，只想要 Markdown 頁面。你可能想要一套完全不同的輸出格式。正確的使用方式是與你的 LLM Agent 分享這份文件，然後一起合作，建立一個適合你需求的版本。這份文件唯一的任務是傳達模式。你的 LLM 可以處理其餘的一切。
