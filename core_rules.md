# Core Rules

所有 AI 代理（Claude Code、Codex、Gemini、其他 coding agents）都應遵守本檔案的共同規則。

## 工具使用優先順序

- 操作 vault 筆記時，優先使用 obsidian-cli skill。
- 只有在 obsidian-cli 無法處理時，才退回使用直接檔案讀寫。

## 語言與語氣

- 預設使用繁體中文回覆與撰寫筆記。
- 技術名詞、工具名稱、API 名稱、程式語言關鍵字可保留英文。
- 回覆要直接、清楚、簡潔；不要過度解釋已經很明顯的內容。
- 如果使用者的需求不明確，先提出少量關鍵問題，不要自行大量假設。

## Vault 用途

這個 vault 主要用於：

- 個人學習筆記
- 工作專案紀錄
- 網頁剪藏與資料整理
- Obsidian、Claude Code、AI skills、工具與插件研究
- 程式設計、資料結構、演算法與系統設計學習

## Obsidian 筆記原則

- 優先使用 Obsidian Flavored Markdown。
- 內部筆記連結使用 wikilinks，例如 `[[Note Name]]`。
- 外部網站使用標準 Markdown link，例如 `[title](https://example.com)`。
- 適合時補上 frontmatter，但不要為了形式而加入無意義欄位。
- 適合時使用 callouts、表格、條列、Mermaid 或 Canvas，但不要過度裝飾。
- 筆記應該方便日後回顧，而不是只做一次性摘要。

## 寫作偏好

- 段落不要太長，優先使用短段落與條列。
- 預設使用精簡條列輸出，不加冗餘說明。
- 除非使用者要求，不主動展開背景說明、動機或補充。
- 不要過度解釋操作步驟，直接執行並簡短說明結果。
- 不要自行大量假設需求，遇到不明確的指令先問 1 個關鍵問題。
- 對學習筆記，建議包含：核心概念、重點整理、例子、常見誤區、延伸連結。
- 對剪藏文章，建議包含：摘要（3句內）、關鍵觀點（條列）、可行動項目；不要逐段翻譯或複製原文，應提煉重點。
- 對工作專案，建議包含：背景、目標、目前狀態、待辦事項、風險、決策紀錄。
- 不要只是美化排版；應該協助提煉、重組與連結知識。

## 資料夾理解

- `Clippings/`：網頁剪藏與外部文章，通常需要整理成摘要、永久筆記或研究筆記。
- `資料結構-鐵人挑戰-35D/`：資料結構與演算法學習筆記，偏教學與系統化整理。
- `工作專案/`：工作、專案、需求、技術紀錄與任務追蹤。
- `obsidian相關筆記/`：Obsidian、Claude Code、skills、插件與 vault 管理相關內容。
- `.claude/skills/`：本 vault 使用的 Claude skills。修改前先理解既有 skill，避免重複建立功能。

## 檔案操作規則

- 優先編輯既有檔案，只有在有明確需求或合理分類時才新增檔案。
- 不要刪除、覆寫或大幅重構筆記，除非使用者明確要求。
- 如果不確定筆記該放在哪個資料夾，先詢問使用者。
- 修改 Markdown 時保留 Obsidian 相容語法。
- 不要建立不必要的 README、規劃文件或臨時文件。

## Skills 與代理協作

- 如果任務適合既有 skill，優先使用既有 skill。
- 建立新 skill 前，先檢查 `.claude/skills/` 是否已有相同或相近功能。
- 新 skill 應該補足實際工作流，而不是重複通用 Markdown、Canvas、CLI 等既有能力。
- 針對本 vault 的 skill，應偏向「如何整理、轉化、連結我的筆記」而不是只描述工具語法。

## 隱私與安全

- 不要主動保存或擴散敏感個資、憑證、token、密碼、私密聯絡資訊。
- 不要把 vault 內容上傳到第三方服務，除非使用者明確同意。
- 不要猜測或編造私人資訊。
- 如遇疑似敏感資料，先提醒使用者並謹慎處理。

## Git 與同步

- 這個 vault 是 git repository，但不要主動 commit、push、reset 或執行破壞性 git 操作，除非使用者明確要求。
- 若看到自動備份 commit 或同步紀錄，不要把它當成任務需求。
- 修改前後應尊重使用者既有未提交變更，不要覆蓋他人的工作。
